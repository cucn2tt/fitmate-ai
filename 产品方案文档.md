# FitMate AI — 产品方案文档

## 一、产品定位

**一句话定位**：面向健身入门者的 AI 私人教练 + 营养师，解决「不知道怎么练、不知道怎么吃、吃了记不住」三大痛点。

**目标用户**：有健身意愿但缺乏专业指导的 25-35 岁上班族，训练场地以居家为主。

## 二、用户痛点与解决方案

| 用户痛点 | 具体场景 | 产品方案 |
|---------|---------|---------|
| 训练盲目 | 打开健身 App 不知道选哪个课程，跟着视频做也不知道对不对 | AI 根据个人画像（目标/水平/场地/时间）每日生成专属训练计划 |
| 饮食失控 | 知道要控制热量，但不会算也不知道吃多少 | AI 基于 BMR/TDEE 精准计算热量缺口，生成可执行的三餐方案 |
| 记录低效 | 拍照记、手写记、App 记，操作繁琐且记了没用 | 拍照自动识别食物 + 热量，AI 对比今日预算给出即时反馈 |
| 数据无感 | 记录了体重腰围，但不知道变化趋势意味着什么 | AI 分析 7 天数据趋势，给出可读的健康洞察 |

## 三、功能架构

```
FitMate AI
├── 每日记录      P0  手动录入体重/腰围
├── 训练计划      P0  AI生成个性化训练（动作+组数+次数）
├── 饮食计划      P0  AI生成热量目标三餐方案
├── 拍照识食      P1  拍照→视觉识别菜品→AI补救建议
├── 体脂分析      P1  AI分析近7日趋势
└── 历史数据      P2  体重/腰围变化折线图
```

### MVP 优先级逻辑

- **P0**：覆盖用户每日核心诉求——练什么、吃什么。这是用户打开 App 的理由。
- **P1**：增强粘性的差异化功能。拍照识食降低记录门槛，体脂分析让数据产生意义。
- **P2**：基础数据可视化，锦上添花。

## 四、核心流程设计

### 4.1 训练计划生成

```
用户画像 → Prompt模板 → DeepSeek LLM → 结构化训练方案
(目标/水平      (健身教练       (热身+正式训练
 /时间/场地)      角色设定)        +拉伸+注意事项)
```

**交互设计**：一键生成，结果即时展示，无需多步操作。每天可重新生成，保证计划不重复。

### 4.2 饮食计划生成

```
用户画像 → 热量计算引擎 → Prompt模板 → AI三餐方案
           (BMR×活动系数  (营养师
            ±热量缺口)     角色设定)
```

**设计要点**：
- 热量计算先于 AI 生成，保证目标可控（不用 AI 来算数学）
- 三餐热量分配：早餐 25% / 午餐 35% / 晚餐 30% / 加餐 10%
- 用户可在侧栏调整饮食限制（如不吃猪肉），实时生效

### 4.3 拍照识食闭环（特色功能）

```
┌─────────┐    ┌──────────────┐    ┌─────────────┐    ┌──────────┐
│ 拍照上传 │ → │ 豆包视觉模型   │ → │ 热量预算对比  │ → │ AI补救建议 │
│         │    │ 识别菜品+热量  │    │ 已吃 vs 剩余  │    │ 晚餐调整   │
└─────────┘    └──────────────┘    └─────────────┘    └──────────┘
```

**用户体验亮点**：不只是「识别这是什么食物」，而是形成「识别 → 计算 → 建议」的完整闭环，即时给出可执行的补救方案。

## 五、用户画像配置

产品将用户特征标准化为可配置参数，训练和饮食方案均基于这些参数个性化生成：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 性别 | 男 | 影响 BMR 计算 |
| 年龄 | 28 | 影响 BMR 计算 |
| 身高 | 175 cm | — |
| 体重 | 70 kg | 每日更新 |
| 目标 | 减脂 | 减脂 / 增肌 |
| 目标体重 | 65 kg | — |
| 训练水平 | 中级 | 新手 / 中级 / 高级 |
| 训练时间 | 30 分钟 | 15-90 分钟可调 |
| 训练场地 | 居家 | 居家 / 健身房 |
| 饮食限制 | 无 | 自由填写，如素食、不吃猪肉 |

## 六、技术选型决策

| 决策点 | 选择 | 选型理由 |
|--------|------|---------|
| 前端框架 | Streamlit | Python 全栈，快速原型验证，无需前后端分离 |
| 语言模型 | DeepSeek | 性价比高，中文能力强，支持类 OpenAI SDK |
| 视觉模型 | 豆包（火山方舟） | 国内可用，食物识别准确，API 调用简单 |
| 数据存储 | CSV | MVP 阶段无需数据库，单用户场景足够 |
| 部署平台 | Streamlit Cloud | 免费，GitHub 联动，一键部署 |

**核心原则**：MVP 阶段追求开发效率最大化，每个选型都在 1 天内可跑通验证。

## 七、产品指标

| 指标 | 目标 | 衡量方式 |
|------|------|---------|
| 核心功能可用率 | 6 个模块均可正常使用 | 功能验收 checklist |
| 训练方案满意度 | 用户连续 3 天主动使用 | 自用观察 |
| 饮食热量准确度 | AI 三餐总热量偏差 < 10% | 手动核算对比 |
| 识食准确率 | 常见中餐菜品识别成功 | 10+ 样本测试 |
| 每日使用耗时 | < 3 分钟完成全流程 | 实际计时 |
